# catch-management モバイルUI仕様（v1ドラフト）

作成日: 2026-07-12
対象: `catch-management`（Next.js + FastAPI + Supabase）モバイルUI
関連: `fishing-dashboard`（Vite + React + deck.gl + Plotly / GitHub Pages）

---

## 0. 前提と役割分担

本仕様は「スマホ = 記録、ブラウザ = 分析」の疎結合アーキテクチャを前提とする。

| 領域 | 担当 | 責務 |
|---|---|---|
| 入力・記録 | catch-management（本仕様の対象） | 釣行中の釣果登録、環境データ自動付与、ポイント管理、マイページでの簡易的な実績表示 |
| 分析・可視化 | fishing-dashboard | クロスフィルタマップ、時系列、相関エクスプローラ等（issue #65 の設計を維持） |

**設計原則**

1. **全魚種対応**。シーバス（多摩川汽水域）は現在の主対象・ワークド例であり、スコープ制約ではない。魚種はユーザーが追加可能なマスタとして扱う。
2. **分析機能はモバイルに持たせない**。グラフ・集計マトリクス等は dashboard の責務。ただしマイページに**数字のみの簡易実績表示**（§6）は置く。
3. **テーマは Tokyo Bay sunset を標準**（背景 `#060d1c`、coral / sun / gold アクセント）としつつ、**テーマ切替可能な構造**とする（ライトテーマは釣りに合う色調で別途設計、§7.1）。`fishing-dashboard/src/theme.ts` の定数と共通トークン化する（§7）。Material標準配色（#1976D2 等）の提案は不採用。
4. **基本はプライベートな操作感**。個人の釣果向上が目的。他アングラーとの交流・情報交換は将来のオプトイン機能とし、希望する人だけが有効化する（§5 リサーチの将来拡張）。
5. ChatGPT提案から採用するのは **入力最小化（3タップ登録）** と **マップのボトムシート**、および **レイヤートグル** と **釣れそう指数（条件付き）** の4点。

## 1. ナビゲーション構成

下部タブ4つ + 中央FAB（登録）。分析タブは置かない。

```
[ 🏠 ホーム ] [ 🗺 マップ ] [ ➕ 登録(FAB) ] [ 🔍 リサーチ ] [ 👤 マイページ ]
```

- **「リサーチ」**（旧: 履歴）: 「履歴」では目的が過去記録の閲覧に限定され拡張性がない。「リサーチ」とすることで、自分の釣果履歴に加え、将来的にアングラー・タックル・別地域などの文脈別の調べ物や、（希望者のみの）アングラー同士の交流・情報交換へ画面を拡張できる（§5）。
- **「マイページ」**（旧: 設定）: 設定はマイページ内の一項目とする。プライベートな情報（実績・自分のデータ・各種設定）を扱う場所であることをイメージしやすくする（§6）。
- FABは常時表示・最大サイズ。釣行中に最も押すボタンのため、親指到達圏（画面下部中央）に固定。
- タブバーは選択中テーマに追従。ダークテーマ時は `#060d1c` 系、アクティブタブはアクセント色。

## 2. ホーム

「今日の状況を10秒で把握する」画面。カード縦積み構成。

### 2.1 コンディションカード（最上部）

- 表示対象ポイント: 最後に使用したお気に入りポイント（未設定時は現在地）。タップでポイント切替。
- ヘッダ行の構成: **ポイント名 / 現在時刻（24h表記）/ 日付** の順。現在時刻を明示することで、潮位グラフの現在時マーカーによる「大体の時間」把握に加え、行動指針の基準時刻を即読みできるようにする。
- 表示項目: 天気アイコン / 気温 / 風向・風速 / 気圧（変化傾向 ↑↓ 付き）。
- **潮位表示（詳細化）**:
  - 現在の潮位（数値cm）+ **上げ / 下げの別**（例:「142cm ▲上げ」）。
  - **当日の満潮・干潮それぞれの時刻と潮位**（例:「満潮 04:12 / 168cm・干潮 10:45 / 38cm」。1日2回転の日は2組表示）。
  - 潮位ミニグラフ（当日24h、現在時刻マーカー、満干時刻ラベル）は継続。数値表示とグラフで、潮位変動から狙うタイミング・ポイント移動・魚の行動予測の材料を可視化する。
- 潮回り（大潮〜長潮）。
- データソース: §8.2 の気象・潮汐APIをサーバー側でキャッシュして配信。

### 2.2 釣れそう指数カード（条件付き表示）

- 対象ポイント × 対象魚種の自分の過去釣果から算出する簡易スコア（★1〜5）。
- **データ充足性ゲート必須**: 該当条件の釣行サンプルが閾値（初期値 20釣行、要調整）未満の場合はスコアを出さない。少サンプルでの断定表示は行わない。
- **充足前の表示は馴染みのある日本語表現とする**（「n=12」「n≧20」等の記号表記は使わない）。例:「データをためています（いま 12回分・あと 8回分の釣行記録で表示できます）」。
- 根拠の内訳（潮回り一致 / 風向一致 / 時間帯一致 等）を展開表示できるようにし、ブラックボックス化しない。
- アルゴリズム詳細は dashboard 側の相関分析ロジックと整合させる（§9 未決事項）。

### 2.3 最近の釣果カード

- **2列グリッド表示**（1列リストはスペース効率と一覧性が低いため不採用）。各セル: サムネイル / 魚種 / サイズ / 日付。
- 日付は**年月日表記**（例: 2026/07/05）。相対表記（「昨日」「3日前」）は使わない。
- **月見出し付き**: グリッドの上に月ラベル（例:「7月」）を置き、月が変わる位置に次の月見出し（「6月」）を挿入してその下に続きのグリッドを並べる。直近の把握を月単位で直感的にできるようにする。
- 表示件数は直近2ヶ月分・最大6件程度（要調整）。セルタップで釣果詳細（§5.3）へ。

### 2.4 ホーム最下部の枠（旧: 分析への導線）

- **fishing-dashboard への外部リンクカードは廃止**。dashboard はPC閲覧前提であり、スマホで開いても見づらく導線の価値がないため。
- 空いた枠の代替候補（検討中、§9）:
  - **月間潮汐カレンダーへのショートカット**: 「今週の大潮はいつか」を先読みして釣行計画に使う。既存の潮汐データの再利用で実現でき、記録アプリの責務内。
  - **前回釣行カード**: 「前回の釣行から n日・前回: シーバス 68cm @六郷橋下流」。釣行間隔の意識づけと記録の連続性。
  - **今日の残り満干カウントダウン**: 「次の満潮まで 2時間14分」。コンディションカードの潮位詳細と重複気味のため優先度低。

## 3. マップ

フルスクリーン地図 + ボトムシート。カードUIは使わない。

### 3.1 地図本体とピン・ラベル表示

- 全画面表示。現在地 / 自分の釣果ピン / お気に入りポイント（★）。
- **魚種の識別は色・アイコンではなく呼称（テキストラベル）で行う**。魚種マスタはユーザー追加可能=無制限のため、色分け（識別可能な色数を超過、凡例依存、色覚配慮の困難）もアイコン分け（小サイズで判別不能、新魚種のアイコン用意が不可能）も全魚種対応と構造的に矛盾する。色は識別手段としては使わない。

**ズーム段階制（POIラベル方式）**

| ズーム | 表示 |
|---|---|
| 低（広域） | クラスタ円 + 件数のみ |
| 中 | 単色ドットのみ（密度把握が目的、個体識別はしない） |
| 高 | ピン + 呼称ラベル |

**ラベル縮退ルール**: 表示中の魚種が2種以上ならラベルは「魚種名 + サイズ」（例:「シーバス 68」）。1種のみならサイズだけ（例:「68」）に縮退。ラベルが識別に必要かどうかだけで内容が決まるため、ユーザーが覚えるルールはない。

**ラベル衝突回避**

- 位置ずらし: 定位置（ピン右横）が重なる場合、左→上→下の候補位置を順に試す。
- 優先度カリング: どこにも置けないラベルは非表示にする。**ピン（ドット）自体は必ず残り、消えるのはテキストのみ**。ズームインで再表示される段階開示。
- 優先度はサイズ大きい順 → 新しい順。同一ズーム・同一データで常に同じ結果になる決定的な順序とする（パン操作でのチラつき防止）。

**魚種チップ（釣果レイヤー内の絞り込み）**

- 地図上部に横スクロールのチップ行。先頭「すべて」固定、以降は釣果件数の多い順。末尾「その他 ▼」でボトムシートから全魚種リスト。
- **複数選択可**（「シーバス + マゴチ」の比較閲覧を想定）。「すべて」タップで全選択に戻る。
- チップは釣果レイヤー内のフィルタであり、レイヤーパネルの釣果ON/OFFとは別階層。釣果レイヤーOFF時はチップ行ごと非表示。
- 選択状態はラベル縮退ルールと連動する（1種のみ選択→サイズのみ表示）。

### 3.2 レイヤートグル（ChatGPT提案から採用）

右上のレイヤーボタンからON/OFF:

| レイヤー | v1 | データソース候補 | 備考 |
|---|---|---|---|
| 自分の釣果 | ✅ | Supabase | 魚種フィルタ付き |
| お気に入りポイント | ✅ | Supabase | |
| 潮汐（テキストバッジ） | ✅ | 気象庁 / 海上保安庁 | 現在潮位・潮回りをオーバーレイ表示 |
| 水深・海底地形 | 🔜 v2 | 海しる（海洋状況表示システム）、ENC由来データ | **配信形式・ライセンス確認が未了**（§9） |
| 航路 | 🔜 v2 | 同上 | 同上 |
| 潮流ベクトル | 🔜 v2+ | 海上保安庁潮流推算 | 表示負荷とAPI可用性を要検証 |

- v1は確実に出せる3レイヤーに絞り、海しる/ENC系はライセンス確認後にv2で追加。

**トグルの挙動（全レイヤー共通）**

- トグルは純粋な**表示/非表示スイッチ**。データ・フィルタ状態・操作系（ボトムシート、「ここで登録」等）には一切影響しない。再ONで即復元。
- トグル状態は**永続化**する（アプリ再起動後も維持）。
- **自動再ON**: 非表示レイヤーに属する物を新規作成した場合、そのレイヤーを自動的にONへ戻す（例: お気に入りOFF中に地図長押しでお気に入り追加 → 追加した瞬間にレイヤー再ON。釣果OFF中に新規登録して地図に戻った場合も同様）。「追加したのに見えない」混乱を防ぐ。

### 3.3 ボトムシート（ChatGPT提案から採用）

画面下端から地図の上にせり上がるドラッグ可能なパネル。地図の文脈を保ったまま詳細を段階開示する。ドラッグして指を離すと最寄りの定位置に吸着する**3段階スナップ**とし、各段階を情報の深さに対応させる。

| 段階 | 高さ目安 | 役割 |
|---|---|---|
| Peek | 約20% | 「何をタップしたか」の確認 + 最重要アクション。地図はほぼ全面見える |
| Half | 約50% | リスト・サマリの流し読み。ピン位置もまだ見える |
| Full | 約90% | 全履歴の閲覧。実質的な詳細画面 |

**ピンタップ時の挙動**

1. Peek で出現: ポイント名 / 釣果件数 / 直近1件の要約 / **「ここで登録」ボタン**。登録ボタンは Peek 時点で表示し、ドラッグなしで届くこと（釣行中の最頻アクションのため）。同時に地図カメラがパンし、ピンがシートに隠れない位置へ移動。
2. Half: そのポイントの**現在の環境サマリ**（潮回り・潮位傾向・風・天気。ホームのコンディションカードと同一データソース）+ 直近釣果リスト。環境サマリを Peek に置かないのは意図的（Peek は確認 + 登録ボタンのみに絞り高さを最小化）。
3. Full: そのポイントの全釣果を魚種チップで絞り込みつつスクロール閲覧。行タップで釣果詳細（§5.3）へ遷移。
4. 閉じる: 下スワイプ or シート外の地図タップ（タップで即座に地図操作へ復帰）。

**地図長押し時の挙動**

- **Peek のみで Half/Full を持たない**（未記録地点のため見せる履歴が存在しない）。
- 長押しで仮ピン（グレー）を設置し、Peek に座標 + 解決された潮汐基準港を表示。アクションは「★お気に入り追加」「ここで登録」の2つ。
- 「お気に入り追加」→ 名前入力の小モーダル → 保存で仮ピンが★に変化（お気に入りレイヤーOFF時は自動再ONが発動）。
- 「ここで登録」→ 登録画面へ、GPS ではなく**長押し座標をプリセット**して遷移（「立ち位置と釣れた場所が違う」ケースに対応。橋上から下流ポイントを指定する等）。
- 仮ピンはシートを閉じると消える一時オブジェクト。
- **近接優先（誤爆防止）**: 長押し地点の近傍（初期値 50m、要調整）に既存の釣果ピン・お気に入りポイントがある場合、仮ピンは立てず**既存ポイントのシート（ピンタップ時挙動）を開く**ことを優先する。

**その他**

- deck.gl移植は不要（マップはモバイル側では軽量なタイル地図 + マーカーで十分。deck.gl の重い可視化は dashboard の責務）。

## 4. 釣果登録（最重要画面）

**目標: 最短3タップで保存**（魚種タップ → サイズ確定 → 保存）。残りは自動付与と任意入力。

### 4.1 入力フロー

```
Step 1  魚種選択    最近使った順の大ボタングリッド（2列 × 4）
                    +「その他」で全魚種マスタ検索
Step 2  サイズ      cm ホイールピッカー（前回値付近を初期位置）
                    + 直接入力用テンキー切替（補助入力、下記）
Step 3  保存        大ボタン（画面下部固定）
```

- **ステップインジケータ**: ヘッダの「釣果を登録」タイトル部に現在ステップを表示（例:「釣果を登録 1/3」+ 進行ドット）。現在地と残り工程を直感的に把握できるようにする。
- **魚種選択は実画像 + 名称**: 選択ボタンの背景に魚種の実画像を敷き、名称を重ねて表示する。イラストアイコンは種の判別がしにくい場合があるため使わない。画像は魚種マスタに紐づけ、プリセット魚種は同梱画像、ユーザー追加魚種は本人の釣果写真から自動設定（未設定時は名称のみのプレーン表示）。
- **テンキー切替の位置づけ（補助入力）**: 主入力はホイールピッカー。テンキーは以下のケースの救済用として残す。
  - 想定外の大物・外道でホイールの初期位置（前回値付近）から大きく離れた値に飛びたいとき（例: 普段40〜60cmの魚種で91cmを入力）。
  - メジャー計測後に正確な数値を最短で打ちたいとき。
  - ホイールの連続スクロールが苦手な操作環境（グローブ等）のフォールバック。

### 4.2 任意項目（折りたたみ「詳細を追加」内）

ルアー/ベイト（最近使った順サジェスト）、重さ、写真（カメラ直起動）、リリース有無、メモ、ヒットレンジ・アクション等の実験レイヤー変数（dashboard の variable registry と項目名を揃える）。

### 4.3 自動付与（保存時にサーバー側で確定）

| 項目 | 取得元 | 備考 |
|---|---|---|
| GPS座標 | 端末 | 精度低下時は最後の高精度位置 + フラグ |
| 日時 | 端末 | |
| 天気・気温・風向風速・気圧 | 気象API | 保存時刻・座標でスナップショット |
| 潮回り・潮位・満干時刻 | 潮汐API | 最寄り基準港を座標から解決 |
| 水温 | TBD | 信頼できるソース未定（§9）。当面は任意手入力欄 |

### 4.4 現場条件への配慮

- **オフラインファースト**: 電波不安定な釣り場を想定し、ローカルキューに保存 → オンライン復帰時に自動同期。環境データの自動付与は同期時にサーバー側で日時・座標から遡って取得（釣行中の通信に依存しない）。
- **片手・濡れ手操作**: 主要UIを画面下半分に配置。タップターゲット48px以上。スワイプ依存の操作は置かない。
- 保存後はトースト + Undo（10秒）を表示してホームへ戻る。連続登録用に「続けて登録」も併置。

## 5. リサーチ（旧: 履歴）

「履歴」から改名。v1の中身は自分の釣果履歴だが、タブの目的を「調べる」に広げることで、将来的にアングラー・タックル・別地域などの文脈別リサーチや、希望者のみのアングラー間交流・情報交換へ拡張できる画面とする（基本はプライベート、交流はオプトイン。設計原則4）。

### 5.0 v1のスコープと将来拡張

- **v1**: 自分の釣果履歴（以下 5.1〜5.3）のみ。
- **将来拡張（本仕様では設計しない）**: タックル別リサーチ / 地域別リサーチ / アングラー交流（オプトイン）。画面上部にセグメント切替を置ける構造だけ想定しておく。

### 5.1 一覧

- カード縦スクロール（サムネ / 魚種 / サイズ / 日付 / ポイント名）。
- 上部にフィルタチップ: 魚種 / 期間 / ポイント。dashboard のフィルタチップとUI言語を揃えるが、機能は絞り込みのみ（集計はしない）。

### 5.2 検索

- 魚種名・ポイント名・メモのテキスト検索。

### 5.3 詳細画面

- 写真、基本情報、**自動付与された環境データの全項目**（天気 / 気温 / 風 / 気圧 / 潮回り / 潮位 / 時間帯）。
- 編集・削除。環境データは「再取得」ボタンで日時・座標から再解決可能（誤記録の修正用）。
- 「この条件で分析 → dashboard」リンク（該当レコードにディープリンクできるかは dashboard 側のURLパラメータ設計次第、§9）。

## 6. マイページ（旧: 設定）

「設定」から改名し、設定はマイページ内の一項目とする。実績・自分のデータ・各種設定というプライベートな情報を扱う場所として構成する（基本はプライベートな操作感。設計原則4）。

### 6.1 実績サマリ（確定・プロトタイプ済み）

現在の「累計釣果数」1行に代えて3行構成とする。いずれも全魚種の合算値。

| 行 | 内容 |
|---|---|
| 今年 | 今年1/1〜今日までの釣果数（ヒーロー表示） |
| 去年同時期 | 去年1/1〜去年の同月同日までの釣果数。括弧内に去年の年間合計を補足（例:「51匹（年間 98匹）」） |
| 累計 | 全期間の釣果数 |

**集計ルール**: 区切りは暦年（1月始まり。年度=4月始まりは不採用）。「同時期」の終端は去年の同月同日、うるう年の2/29は2/28として扱う。日付境界はJST、集計対象は `recorded_at`。

### 6.2 魚種別実績カード（確定・プロトタイプ済み）

魚種別自己ベストを2行構成のカードに拡張する。

- 1行目:「{魚種名}」（左）/「自己ベスト {n}cm」（右）
- 2行目:「今年 {n} ・ 去年同期 {n} ・ 累計 {n}」（集計ロジックは §6.1 と同一を魚種で絞って適用）
- 並び順は累計匹数の降順。デフォルトは上位5魚種のみ表示し、「すべて表示」で全魚種に展開。

**エッジケース**: 去年データ0件は「—」（年間補足も省略）。記録開始が今年の場合も去年行は「—」で残す（レイアウト安定のため）。サイズ未入力のみの魚種は「自己ベスト —」。

**スコープ**: マイページの実績はサマリ3行と魚種別実績カードのみ。年推移グラフ・魚種別の年間内訳マトリクスは追加しない（分析は dashboard の責務）。

プロトタイプ: `docs/prototypes/settings-stats-prototype.html`

### 6.3 管理・データ・設定

- **お気に入りポイント管理**: 追加 / 名称編集 / 並べ替え / 削除。潮汐基準港の手動上書き。
- **魚種マスタ管理**: 全魚種対応の中核。プリセット + ユーザー追加。表示順は使用頻度順。魚種画像の設定（§4.1 の実画像選択に使用）。
- **ルアー/ベイトマスタ管理**: 同上。
- **データ**: エクスポート（dashboard 連携フォーマット）、同期状態表示、オフラインキューの中身確認。
- **表示設定**: 単位（cm/inch は当面 cm 固定でも可）、**テーマ切替（ダーク / ライト、§7.1）**、ナイトモード（§7.1）。

## 7. 共通UI原則

### 7.1 テーマ

- **標準テーマは Tokyo Bay sunset（ダーク）**。夜釣り前提の実用性とブランド一貫性から、初期値・デフォルトはこちら。
- **テーマ切替を可能にする**（マイページ > 表示設定）。ライトテーマは「釣りに合う色調」で別途設計する — 方向性としてはデイゲームの水辺（明るい空・浅瀬の砂色・水面のブルーグリーン等）を候補とし、具体配色は未決（§9）。
- 切替を前提とするため、全コンポーネントは**セマンティックなデザイントークン**（`--bg` / `--surface` / `--accent` 等）経由で配色し、生のHEX直書きを禁止する。
- `fishing-dashboard/src/theme.ts` の色定数を共通デザイントークン化する。方法は (a) 共有パッケージ切り出し、(b) 単一ソースからの生成、(c) 複製 + 差分チェックCI、のいずれか（§9）。dashboard 側は Tokyo Bay sunset のみで切替なし（PC分析画面のため）。
- **ナイトモード（追加検討）**: ダークテーマよりさらに輝度と彩度を落とした釣行中モード。暗所での眩惑防止。テーマとは独立したモディファイアとして扱う。

### 7.2 コンポーネント使い分け（ChatGPT提案の使い分け方針を採用）

| 目的 | パターン |
|---|---|
| 情報のまとまりを見せる | カード（ホーム、リサーチの釣果一覧） |
| 入力 | 縦一列フォーム + 大ボタン（登録） |
| 地図 | フルスクリーン + ボトムシート |
| 文脈操作・詳細 | ボトムシート / モーダル |

## 8. データ・API面の含意

### 8.1 Supabase スキーマへの影響

- `catches` に環境スナップショット列（weather, air_temp, wind_dir, wind_speed, pressure, tide_type, tide_level, water_temp nullable, ...）。生APIレスポンスの `raw_env` JSONB も保持し、後から変数を再抽出可能にする（dashboard の variable registry / 実験レイヤーとの整合）。
- `species`（魚種マスタ、user scope + preset）、`points`（お気に入りポイント、潮汐基準港ID付き）、`lures`。
- オフライン同期用: クライアント生成UUID + `recorded_at` / `synced_at` 分離。

### 8.2 外部API候補

- 気象: 気象庁 or Open-Meteo（キー不要・座標指定可でプロトタイプ向き）。
- 潮汐: 気象庁潮位表 / 海上保安庁。基準港解決ロジックが必要。
- 海しる / ENC レイヤー: ライセンス・配信形式（WMS/タイル）の調査が前提。

### 8.3 fishing-dashboard 連携

- 既存の read-only データパイプラインを維持。本仕様で増える環境スナップショット列は dashboard の相関エクスプローラの変数として自動的に価値を持つ。
- 新規変数追加時は variable registry への昇格フローに乗せる（既存設計どおり）。

## 9. 未決事項

1. 水温の信頼できるデータソース（沿岸・河口域でメッシュが粗い問題）。
2. 海しる / ENC 由来レイヤーのライセンスと配信形式。
3. 釣れそう指数のアルゴリズムと充足性閾値（dashboard の効果量計算と整合させるか、独立の簡易スコアにするか）。
4. テーマトークン共有の実装方式（共有パッケージ / 生成 / 複製 + CI）。
5. dashboard へのディープリンク仕様（レコードID・フィルタ状態をURLパラメータで渡せるか。§5.3 の詳細画面リンクの要否含む）。
6. 登録画面の実験レイヤー変数をどこまで任意項目に出すか（出しすぎると入力最小化と衝突）。
7. **ライトテーマの具体配色**（「釣りに合う色調」の設計。デイゲームの水辺を方向性の候補とする）。
8. **ホーム最下部枠の中身**（§2.4 の代替候補: 月間潮汐カレンダーショートカット / 前回釣行カード / 満干カウントダウン）。
9. **プリセット魚種の実画像の調達**（同梱画像のライセンス。自前撮影 / パブリックドメイン図鑑等）。
